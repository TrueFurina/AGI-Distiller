# 调试 5 步流程

用户问题: $ARGUMENTS

## 规则

1. **严格按 5 步走，不许跳过任何一步。**
2. 步骤 1-2 只读不改——未确认方案前不修改任何文件。
3. 步骤 3 只做最小改动——不许顺手重构、不许改无关模块、不许格式化无关文件。
4. 步骤 4 先补会失败的用例，再改代码让它通过。
5. 步骤 5 输出回归表，区分"必须通过"和"低风险抽查"。

---

## 步骤 1：复现

> 调试的起点不是"哪里错了"，而是"怎样稳定看到错误"。

### 输出格式

```
## 复现说明
- 输入/操作路径：
- 预期结果：
- 实际结果：
- 报错信息/日志：
- 复现成功率（必现/偶现/环境相关）：
- 还缺什么信息（如果有）：
```

### 执行

```bash
# 如果用户提供了报错信息，先尝试定位关键文件
echo "尝试查找报错中提到的文件引用..."
# 不做修改，只读搜索
```

---

## 步骤 2：定位（只读，不改代码）

> 缩小范围：只读相关模块、只解释调用链、只列候选原因。

### 输出格式

```
## 定位分析
- 可疑文件/模块：
- 调用链分析：
- 候选原因（按可能性排序）：
  - 原因 1：<证据>
  - 原因 2：<证据>
- 排除的原因及理由：
```

### 约束

- 不许修改文件
- 不许一次性搜索全仓库——从用户指明的范围开始
- 如果原因不明，如实说明"暂无法确定"，给出排查思路

---

## 步骤 3：最小改动

> 只做最小改动，不许顺手重构。改完后说明三个问题。

### 输出格式

```
## 改动说明
- 修改了哪些文件：
- 每个文件改了什么：
- 为什么这些改动能覆盖复现路径：
- 可能影响的相邻功能：
- 本次特意没有改什么（及其原因）：
```

### 约束

- 不许改无关模块
- 不许重构
- 不许格式化未修改文件
- 不许为了"看起来更干净"调整代码结构

---

## 步骤 4：测试

> 先补最小可运行用例（会失败的），再改代码让它通过。

### 输出格式

```
## 测试验证
- 测试用例覆盖了什么场景：
- 运行命令：
- 结果：
- 如果失败：
  - ❌ 业务代码 bug → 继续修
  - ❌ 测试断言写错 → 改测试
  - ❌ 环境问题 → 修运行方式
```

### 执行

```bash
# 运行项目已有测试
if [ -f "Cargo.toml" ]; then
  cargo test 2>&1 | tail -30
elif [ -f "package.json" ]; then
  npm test 2>&1 | tail -30
elif [ -f "pyproject.toml" ] || [ -f "requirements.txt" ]; then
  python -m pytest --tb=short 2>&1 | tail -30
fi
```

---

## 步骤 5：回归

> 修完 Bug 后，验证"没坏掉的东西"。

### 输出格式

```
## 回归检查
### 必须通过
- 原 Bug 复现路径已修复：
- 核心功能是否正常：

### 高风险相邻功能
- 同一模块的核心流程：
- 可能受影响的入口：

### 低风险抽查
- 页面渲染/日志/权限/异常提示：
- 不在本次范围（已记录）：
```

---

## 最终输出结构

```
# 调试报告

## 问题
[一句话描述]

## 根因
[为什么出错]

## 改动
[改了哪些文件、怎么改的]

## 验证
[测试结果]

## 风险
[还有哪些需要注意的]
```
